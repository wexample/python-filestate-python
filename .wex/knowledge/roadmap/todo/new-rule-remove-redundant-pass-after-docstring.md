# New rule: remove redundant `pass` after a docstring

**Source**: *(new option — `option/python/remove_redundant_pass_option.py`)*
**Agent**: agent:rectifier-rules
**Bucket**: new-rule
**Severity**: feature

## Symptom
Common pattern in our code (and in autogenerated stubs):

```python
class Foo:
    """Doc."""
    pass

def bar():
    """Doc."""
    pass
```

The `pass` is dead — the docstring is already a valid suite body. Black does not remove it. Same for empty `else: pass` / `except: pass` (the latter is *also* a smell — see sibling ticket on bare except).

## Suggested direction
LibCST visitor that, for any `cst.FunctionDef` / `cst.ClassDef` / `cst.AsyncFunctionDef` body containing exactly two statements `[SimpleString (docstring), Pass]`, removes the trailing `Pass`. Same for `cst.If.orelse` / `cst.Try.handlers` bodies where the only statement is `Pass` and a docstring precedes it (rarer but valid). Use the cached module from `cst_cache`. Idempotency check is trivial: walk once, return `src` if no candidate found.

Out of scope: stub `.pyi` files (where `pass`-bodied classes are sometimes intentional placeholders — exclude by suffix).
